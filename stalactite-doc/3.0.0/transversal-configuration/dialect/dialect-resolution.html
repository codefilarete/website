Dialect resolution

ServiceLoaderDialectResolver manages the way that Dialects are found and chosen for a database connection. As mentionned in Dialect registration paragraph, Stalactite will use the JVM Service Provider system to find the available Dialects, then it will select the “best matching” one according to the database version (took from JDBC Connection metadata) and the database compatibility given by DialectResolverEntry : vendor name must match exactly and the closest version is kept, with preference for smaller one. For example, with HSQLDB 2.5.2 (took from database connection), an entry giving HSQLDB 2.5.0 will be preferred to a HSQLDB 3.0.0 one.

If this algorithm doesn’t suit your need, one can change it by implementing your own ServiceLoaderDialectResolver and give it as an argument to PersistenceContext at instantiation time.

By default, PersistenceContext will auto-discover the adequate Dialect when you pass it a DataSource as a constructor argument.
